# 软件组合功能蓝图

> 版本：v1.0  
> 更新日期：2026-03-25  
> 目的：基于当前数据资产和已有设计文档，重新梳理软件的组合功能，形成开发基线。

---

## 1. 产品定位

本项目不是通用 BI 平台，也不是节能推荐平台的完整业务系统。  
它的核心定位应当是：

**面向建筑能耗时序建模的 XGBoost 实验工作台。**

它服务的是“从数据资产到实验结果”的完整链路，重点在：

1. 数据资产登记与理解。
2. 特征工程与切分策略配置。
3. XGBoost 训练过程可视化。
4. 结果评估、对比与复现实验。

---

## 2. 设计边界说明

`docs/design/references/` 中的文档和截图仅可作为视觉与交互灵感来源，**不得直接照搬为项目实现**。

原因如下：

1. 其中包含“节能推荐”“行业对标”“登录页”等业务页面，超出当前数据与开发阶段的核心目标。
2. 其中的页面信息架构是参考原型，不等于当前项目的实现基线。
3. 当前项目应优先围绕真实数据集构建研发工作台，而不是先实现业务展示外壳。

因此，功能规划应以 `dataset/` 中的真实数据资产为主，以 `docs/design/references/` 为次要参考。

## 3. 软件应具备的组合功能

软件建议按“一个主流程、七个组合模块”来组织。

### 3.1 主流程

```text
数据资产登记
-> 数据理解与质量检查
-> 特征工程与切分
-> 实验配置
-> XGBoost 训练
-> 训练监控
-> 结果评估与实验对比
```

### 3.2 组合模块

#### 模块 A：数据资产工作台

目标：

- 让用户从“仓库已有数据”或“新增本地文件”中创建可用数据资产。

功能：

- 扫描并登记 `dataset/` 目录中的现有数据集。
- 支持单文件、文件夹、多个 CSV 文件集合三种导入方式。
- 识别数据源类型、时间范围、采样粒度、实体粒度。
- 标记原始/清洗状态。
- 支持把“多个 CSV 共同构成一个逻辑数据集”登记为一个数据资产。
- 支持声明文件成员角色，如主表、天气表、元数据表、补充分片。

适配数据：

- `HEEW`
- `ashrae-gepiii`
- `bdg2`
- `Bldg59`
- `building_energy_data_extended.csv`

#### 模块 B：模式映射与数据质量检查

目标：

- 在训练前把“可用性”和“可解释性”问题暴露出来。

功能：

- 时间列、实体列、目标列映射。
- 数值列/分类列识别。
- 缺失值、重复值、异常值、时间连续性检查。
- 原始表与天气表/元数据表的关联校验。
- 生成可复用的 schema profile。

#### 模块 C：特征工程与数据切分

目标：

- 让数据从“可读”变成“可训练、可复现”。

功能：

- 时间特征：小时、星期、月份、节假日占位。
- 滞后特征、滚动统计特征。
- 天气融合、元数据融合、占用特征融合。
- 重采样与时间对齐。
- 时间切分、随机切分、建筑留一、站点留一等评估方案。

#### 模块 D：实验配置中心

目标：

- 让一次实验成为可保存、可复现、可比较的配置对象。

功能：

- 选择数据资产与特征方案。
- 配置目标列、训练集/验证集/测试集方案。
- 配置 XGBoost 参数。
- 保存实验草稿与模板。

#### 模块 E：训练执行与过程监控

目标：

- 把训练从黑盒变成可观测过程。

功能：

- 启动、排队、停止实验。
- 显示训练状态、轮次进度、关键指标变化。
- 实时日志与资源占用摘要。
- 失败原因回显与重试入口。

#### 模块 F：结果分析与 benchmark 评估

目标：

- 用统一方式衡量不同实验的表现。

功能：

- RMSE、MAE、MAPE、R2 等回归指标。
- 预测值 vs 实际值、残差分布、特征重要性。
- 不同数据集、不同切分策略下的结果对比。
- 针对 ASHRAE/BDG2 的 benchmark 模式输出。

#### 模块 G：实验资产库

目标：

- 让历史实验变成知识资产。

功能：

- 实验列表、筛选、标签、备注。
- 保存模型文件、结果快照、训练配置。
- 多实验比较。
- 导出结果表和评估报告。

## 4. 功能优先级重排

## 4.1 P0：开发起步必须落地

1. 数据资产登记。
2. 模式映射。
3. 文件夹上传与多 CSV 逻辑数据集登记。
4. 基础质量检查。
5. 时间切分与训练/验证/测试配置。
6. XGBoost 参数配置。
7. 训练启动与进度监控。
8. 基础结果页。
9. 实验列表与结果对比。

## 4.2 P1：形成研发工作台能力

1. 多表 Join 与天气/元数据融合。
2. 可复用特征方案。
3. 建筑留一、站点留一等 benchmark 评估。
4. 更细致的结果分析。
5. 模型与结果导出。
6. Bldg59 多源特征融合支持。

## 4.3 P2：研究增强能力

1. Google Trends 外部特征融合。
2. 迁移实验模板。
3. SHAP 等解释性增强。
4. 数据/模型版本治理增强。

---

## 5. 建议下调或移出的功能

以下能力不是当前开发基线，建议后置：

1. 登录、注册、OAuth。
2. 节能方案推荐。
3. 行业对标大屏。
4. 在线预测服务。
5. 多算法平台化支持。
6. 复杂权限系统。

说明：

- 这些能力部分来自视觉参考原型，但并不由当前数据资产和近期开发目标直接驱动。
- 当前阶段更重要的是把“数据到实验”的研发主流程打通。

## 6. 页面/视图建议

建议把页面先收敛为 8 个核心视图：

1. 数据资产列表
2. 数据资产详情与预览
3. 模式映射与质量检查
4. 特征工程与切分
5. 实验配置
6. 训练监控
7. 结果分析
8. 实验库与对比

这样能覆盖核心工作流，同时避免被参考原型中的外围页面分散精力。

## 7. 用户主路径

### 路径 1：快速原型验证

```text
选择 demo 数据
-> 映射字段
-> 配置目标与特征
-> 启动训练
-> 查看结果
```

### 路径 2：真实数据集实验

```text
登记 HEEW / ASHRAE / BDG2 / Bldg59
-> 检查数据质量
-> 选择切分策略
-> 配置特征工程
-> 启动实验
-> 监控训练
-> 与历史实验对比
```

### 路径 3：研究型增强实验

```text
导入外部特征
-> 构建复合特征方案
-> 运行 benchmark
-> 观察泛化与解释结果
```

## 9. 对开发的直接要求

1. 先做“数据集登记”而不是只做“上传文件”。
2. 上传能力必须支持文件夹上传，并允许一个数据集由多个 CSV 文件组成。
3. 先做“实验对象”而不是只做“一次性训练按钮”。
4. 先做“评估方案配置”而不是只做固定随机切分。
5. 先做“结果可比较”而不是只做单次结果展示。
6. UI 设计应服务研发流程，不应被参考原型牵着走。
