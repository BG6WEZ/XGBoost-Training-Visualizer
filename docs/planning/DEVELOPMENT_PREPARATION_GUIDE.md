# 开发准备指南

> 版本：v1.0  
> 更新日期：2026-03-25  
> 目的：在正式编码前明确开发阶段、技术边界、目录建议与验收重点，降低后续返工成本。

---

## 1. 当前项目状态判断

当前仓库的真实状态是：

1. 已有较完整的数据资产。
2. 已有较多前期设计文档。
3. 当前工作区尚未形成稳定的实现代码基线。

因此，开发准备的重点不是继续扩写概念，而是先建立一套最小、清晰、可执行的实施基线。

---

## 2. 开发原则

### 2.1 数据优先

- 先围绕真实数据集设计产品流程，再确定页面细节。
- 先支持 `dataset/` 中已有数据，再考虑纯上传场景。
- 上传场景必须支持“文件夹上传”和“多 CSV 组合为一个数据集”。

### 2.2 原始数据不可变

- 不直接改写 `dataset/` 下文件。
- 清洗、重采样、Join、特征工程产物统一写入工作目录。

### 2.3 实验可复现

- 每次实验必须保存数据资产版本、切分方案、特征方案、参数配置和结果快照。

### 2.4 设计参考隔离

- `docs/design/references/` 只作灵感输入，不作为直接实现蓝本。

### 2.5 先打通主链路

- 优先完成“数据登记 -> 特征/切分 -> 训练 -> 评估 -> 对比”主链路。

---

## 3. 推荐技术基线

在当前阶段，建议优先采用“少服务、强主链路”的实现方式。

### 3.1 建议架构

1. `web`
   - React + TypeScript + Vite
   - 用于数据工作台、实验配置、训练监控、结果分析

2. `api`
   - Python FastAPI
   - 负责数据资产登记、模式分析、实验管理、结果查询

3. `worker`
   - Python 后台任务进程
   - 负责数据处理、特征工程、模型训练、批量评估

4. `storage`
   - PostgreSQL：元数据、实验记录、结果摘要
   - Redis：队列、任务状态、实时指标缓存
   - 本地文件系统或 MinIO：中间数据、模型文件、导出结果

说明：

- 这样更贴近项目当前的数据与算法重心。
- 若后续需要再拆分 Node API 网关，可作为扩展，不必作为 MVP 前置条件。

---

## 4. 推荐目录规划

建议在正式开发时按以下方式组织实现代码：

```text
apps/
  web/                  # 前端应用
  api/                  # FastAPI 服务
  worker/               # 后台训练与数据任务

workspace/
  data/raw-index/       # 已登记数据资产的元信息
  data/processed/       # 标准化、清洗、Join、特征产物
  runs/                 # 训练运行日志与临时文件
  models/               # 模型文件
  reports/              # 导出报告

docs/
  architecture/
  specification/
  planning/
  design/
```

## 5. 阶段划分

### 阶段 0：工程脚手架

目标：

- 建立前端、API、worker 的最小可运行骨架。

完成标准：

- 前后端可启动。
- API 可返回健康检查。
- 能读取项目工作目录配置。

### 阶段 1：数据资产登记

目标：

- 从 `dataset/` 扫描并登记数据资产。

完成标准：

- 可列出数据资产清单。
- 可展示基础 schema 与时间范围。
- 可选择目标列、时间列、实体列。

### 阶段 2：特征与切分工作台

目标：

- 打通训练前准备链路。

完成标准：

- 支持时间切分。
- 支持基础特征工程。
- 支持生成可复用方案。

### 阶段 3：训练与监控

目标：

- 让实验真正可运行、可观察。

完成标准：

- 可启动 XGBoost 回归训练。
- 可查看状态、日志、关键指标。
- 失败时可回显原因。

### 阶段 4：结果分析与实验对比

目标：

- 让实验结果可解释、可对比。

完成标准：

- 展示基础指标。
- 展示特征重要性与残差分析。
- 支持至少 2 次实验对比。

### 阶段 5：高级数据与 benchmark

目标：

- 支持 ASHRAE、BDG2、Bldg59 的进阶场景。

完成标准：

- 支持多表关联。
- 支持 benchmark 评估配置。
- 支持高频多源数据对齐。

## 6. 第一批开发任务建议

建议先做以下任务，不要一开始就铺满所有页面。

1. 数据集扫描器
   - 扫描 `dataset/` 子目录
   - 生成数据资产索引

2. Schema 探查器
   - 读取 CSV 头部
   - 分析列类型、时间粒度、缺失率

3. 数据资产列表页
   - 展示已登记数据集
   - 可进入详情

4. 数据资产详情页
   - 预览样本数据
   - 配置时间列/目标列/实体列

5. 数据集导入编排器
   - 支持文件、文件夹、多 CSV 集合上传
   - 将多个 CSV 绑定为一个逻辑数据集
   - 支持成员文件角色标记

6. 实验创建页
   - 选择数据资产
   - 设置切分与参数

7. 后台训练任务
   - 读入标准化后的训练数据
   - 运行 XGBoost
   - 写入指标与日志

8. 训练监控页
   - 轮询或实时查看任务状态

9. 结果页
   - 展示指标与图表

---

## 7. 关键数据对象

建议优先设计以下元数据表或等价对象：

1. `dataset_assets`
   - 数据资产主表

2. `dataset_profiles`
   - schema、质量、粒度等分析结果

3. `feature_recipes`
   - 特征工程方案

4. `split_recipes`
   - 切分方案

5. `experiments`
   - 实验定义

6. `experiment_runs`
   - 实验运行记录

7. `run_metrics`
   - 训练和评估指标

8. `artifacts`
   - 模型、图表、导出文件

---

## 8. MVP 验收标准

达到以下条件，可认为已经完成可开发、可演示的 MVP：

1. 能从 `dataset/` 中登记至少 3 类数据源。
2. 能上传一个文件夹并将其中多个 CSV 识别为一个逻辑数据集。
3. 能基于一个已登记数据资产完成一次 XGBoost 训练。
4. 能配置至少一种时间切分策略。
5. 能查看训练结果与特征重要性。
6. 能保存并重新打开历史实验。
7. 能对比至少两个实验结果。

## 9. 风险与提前规避

### 9.1 风险：一开始就追求“大而全”

规避：

- 先围绕 `demo + HEEW + ASHRAE` 打通流程。

### 9.2 风险：把视觉参考当成交付蓝图

规避：

- 每个页面实现前都先对应到真实数据任务和用户动作。

### 9.3 风险：多数据源统一抽象不足

规避：

- 尽早建立 `dataset asset / schema profile / feature recipe / experiment run` 这四层对象。

### 9.4 风险：训练流程不可复现

规避：

- 所有实验必须固化输入数据、切分、特征与参数。
