## 3. 多 Agent 协作规范

### 3.1 协作原则

#### 核心原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个 Agent 只负责一个明确的领域或模块 |
| **接口先行** | 先定义接口类型，再实现具体逻辑 |
| **文档驱动** | 所有决策和变更必须更新文档 |
| **增量开发** | 小步快跑，频繁集成 |
| **禁止假数据** | **不得为了加快开发进度采用假数据走捷径。所有数据必须通过真实API获取，前端不得硬编码任何业务数据。** |
| **阶段验收** | **每个阶段必须所有功能点都完成测试后，才能进入下一阶段开发。未通过测试的功能点必须在当前阶段修复。** |

#### 数据真实性要求

```
┌─────────────────────────────────────────────────────────────┐
│                     数据真实性规范                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ 允许的 Mock 场景：                                       │
│  - 第三方服务的接口 Mock（如发送邮件服务）                   │
│  - 开发环境的基础设施 Mock（如本地数据库连接）               │
│  - 单元测试中的依赖 Mock                                     │
│                                                             │
│  ❌ 禁止的假数据行为：                                       │
│  - 前端页面硬编码业务数据（如实验列表、指标数据）            │
│  - 跳过 API 调用直接返回假数据                               │
│  - 测试用例中使用固定假数据覆盖边界情况                      │
│  - 隐藏或跳过数据验证逻辑                                    │
│                                                             │
│  验收标准：                                                  │
│  - 断开 API 服务后，前端页面应显示加载失败，而非假数据       │
│  - 所有列表数据必须来自数据库查询                            │
│  - 所有图表数据必须来自真实训练结果                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 变更协调机制

#### 变更影响范围矩阵

| 变更类型 | 影响范围 | 需要同步的 Agent | 审批要求 |
|----------|----------|------------------|----------|
| 新增 API 端点 | API, Web | Agent B → Agent A | 更新 API 文档 |
| 修改数据模型 | API, Trainer, Types | Agent B → Agent C, D | 更新 Schema 文档 |
| 修改共享类型 | Types, Web, API | Agent D → Agent A, B | 全员确认 |
| 新增页面/组件 | Web | Agent A | 无需同步 |
| 修改训练逻辑 | Trainer | Agent C | 更新训练文档 |
| 新增数据库表 | API, Types | Agent B → Agent D | 更新数据模型文档 |

#### 变更提交流程

```
1. Agent 提出变更
   ├── 填写变更描述
   ├── 标注影响范围
   └── 提供测试方案

2. 影响评估
   ├── 检查是否影响共享包
   ├── 检查是否需要类型同步
   └── 检查是否需要文档更新

3. 执行变更
   ├── 先更新类型定义 (packages/types)
   ├── 再更新实现代码
   └── 最后更新文档

4. 验证
   ├── 类型检查通过 (pnpm typecheck)
   ├── 测试通过 (pnpm test)
   └── 文档已更新
```
