# 详细开发计划

> 版本：v1.0  
> 更新日期：2026-03-25  
> 说明：该计划基于当前数据资产、功能蓝图和开发准备文档整理，作为正式开发排期基线。

---

## 1. 开发目标

首期目标不是做完整平台，而是完成一套可运行的 XGBoost 实验工作台 MVP，重点打通：

1. 数据资产登记。
2. 文件夹上传与多 CSV 逻辑数据集导入。
3. 数据预览、映射、质量检查。
4. 特征工程与切分。
5. XGBoost 训练与监控。
6. 结果分析与实验对比。

---

## 2. 里程碑

### M0：工程骨架

输出：

- `web/api/worker` 最小骨架
- 配置文件
- 健康检查
- 基础日志

完成标准：

- 前后端可启动
- API 可用
- worker 能消费空任务

### M1：数据资产管理

输出：

- 数据资产扫描器
- 文件上传、文件夹上传、多 CSV 集合导入
- 数据资产列表与详情页

完成标准：

- 能从 `dataset/` 扫描登记资产
- 能上传本地文件夹
- 能将多个 CSV 绑定为一个逻辑数据集

### M2：数据理解与准备

输出：

- schema 探查
- 列映射
- 基础质量检查
- 特征工程与切分配置

完成标准：

- 能配置时间列、实体列、目标列
- 能执行至少一种时间切分
- 能保存特征与切分方案

### M3：训练执行

输出：

- 实验创建
- XGBoost 训练任务
- 任务状态与日志

完成标准：

- 能启动训练
- 能查看进度与日志
- 能保存训练结果

### M4：结果与对比

输出：

- 指标页
- 特征重要性
- 残差分析
- 实验对比页

完成标准：

- 能查看单次结果
- 能对比两次以上实验

## 5. P0 明确要求

### 5.1 数据集上传

必须支持：

1. 单个 CSV 上传
2. 多个 CSV 同时上传
3. 文件夹上传
4. 将多个 CSV 归并为一个逻辑数据集

逻辑数据集示例：

1. `train.csv + weather.csv + metadata.csv`
2. `BN001_energy.csv + BN002_energy.csv + BN003_energy.csv`
3. `part-001.csv + part-002.csv + part-003.csv`

### 5.2 用户可配置能力

用户至少要能指定：

1. 主表文件
2. 辅表文件
3. Join 键
4. 时间列
5. 目标列
6. 实体列

---

## 6. 风险清单

1. 多 CSV 数据集抽象不统一
2. 文件夹上传后成员关系丢失
3. 训练前标准化流程不可追溯
4. 结果页先做出来但输入链路不稳定

对应策略：

1. 先定义 `dataset_asset` 和 `member_files`
2. 上传成功后立即固化文件成员清单
3. 所有转换写入 artifact 元数据
4. 严格按里程碑顺序推进
