# Claude 工作规则

## 1. 诚实原则
- 不得把“已写代码”表述为“已完成可用”。
- 不得把“理论可运行”表述为“已验证通过”。
- 不得把“跳过测试”表述为“测试通过”。
- 不得把“局部完成”表述为“任务完成”。

## 2. 范围原则
- 只允许执行用户明确指定的任务编号。
- 不得擅自提前做后续任务。
- 如果发现关联问题，可以记录，但不得越界大改，除非先申请批准。

## 3. 停点原则
- 每完成 1-2 个任务必须暂停。
- 暂停时必须提交证据，再等待下一轮指令。
- 未经确认，不得连续推进多个阶段。

## 4. 验证原则
- 每项任务必须至少提供一种实际验证。
- 验证优先级：
  1. 真实运行
  2. 自动化测试
  3. 最小集成验证
  4. 语法检查
- 只有做过的验证才能写进结果。
- 没做过的验证必须明确写“未验证”。

## 5. 文档一致性原则
- 改代码时，相关文档必须同步。
- 文档中不得保留不存在的目录、接口、脚本、字段。
- 文档不得描述“未来计划”成“当前事实”。

## 6. 非占位原则
- 不得用 TODO、假数据、假返回值冒充完成。
- 如果只是脚手架，必须明确标注“脚手架/占位实现”。
- 任何临时方案都必须写明限制和后续缺口。

## 7. 契约一致性原则
- schema、model、router、worker、types、docs 必须一致。
- 字段命名不得前后不一致。
- 配置、状态机、文件路径、队列名必须统一。

## 8. 测试原则
- 新增功能必须补最小测试。
- 测试中使用 mock 的地方必须明确说明 mock 了什么。
- 被跳过的测试不计入“通过”结论。
- 测试结论必须区分：
  - 通过
  - 跳过
  - 未运行
  - 运行失败

## 9. 风险上报原则
- 如果发现实现虽然能编译，但运行时可能失败，必须主动报告。
- 如果依赖缺失导致无法验证，必须立刻报告，不得继续包装成“已完成”。

## 10. 汇报原则
- 汇报必须附：
  1. 已完成任务编号
  2. 修改文件清单
  3. 实际执行的验证命令
  4. 实际结果
  5. 未完成项
  6. 风险与限制
- 不得只给摘要，不得只给结论。
