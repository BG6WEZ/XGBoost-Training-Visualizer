## 四、任务范围

### 任务 1：补齐 Worker 可观测性（必须完成）

目标：
- 让 e2e 脚本能稳定感知 worker 可用状态，避免长期 `not_available`。

必须完成（二选一或组合）：
1. 实现 Worker 状态端点（推荐）：
- 新增并暴露可用于探活/状态确认的端点（如 `/health` 或训练状态摘要端点）
- e2e 脚本对接该端点并输出明确状态（healthy/degraded/unavailable）

2. 兼容策略修正：
- 若短期无法提供端点，需在 e2e 脚本中明确将 worker 检查标记为“可选项”并输出风险级别
- 禁止误导性文案（如把缺失端点当作无影响成功）

3. 回归验证：
- 新增或补强测试，覆盖 worker 状态可用与不可用两种场景

验收标准：
- e2e 输出中的 worker 状态语义清晰可诊断
- 至少有自动化测试覆盖该逻辑

---

### 任务 2：完善模型校验语义并固化门禁标准（必须完成）

目标：
- 消除 `model_type: unknown` 等模糊结果，保证模型校验结果可解释、可审计。

必须完成：
1. 模型校验增强：
- 明确模型类型判定规则（如 xgboost json/bin）
- 输出可解释字段：格式、大小、关键元信息是否齐全

2. 门禁标准固化：
- 在 README 中补充“通过/降级通过/失败”的判定标准
- 明确 worker 不可用、模型类型未知等情形属于哪一级别

3. 实测验证：
- 执行 `pnpm test:e2e:results` 和 `pnpm test:e2e:results:json`
- 展示增强后的输出证据

验收标准：
- `model_validation` 输出不再是模糊 `unknown`（除非有明确原因与等级）
- 门禁标准在文档中明确且与脚本行为一致
