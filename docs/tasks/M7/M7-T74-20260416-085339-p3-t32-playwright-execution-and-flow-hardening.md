### 3) 把弱用例补强成真实业务流程验证

当前很多测试只是“打开页面/看标题/截图”，这不够。  
本轮至少需要把以下核心要求落成**真实动作 + 真实断言**：

#### 登录流程

- 正确用户名密码可登录并进入受保护页面
- 错误密码显示明确错误提示
- 登出后重定向到登录页

#### 数据资产流程

- 实际上传一个 CSV 文件
- 实际注册一个数据集
- 注册成功后跳转到详情页并验证详情内容

#### 实验流程

- 实际创建实验（选择数据集、模板、参数）
- 创建后列表中出现该实验
- 实际提交训练任务

#### 结果查看

- 打开已完成实验结果页
- 实际验证指标数据可见
- 实验对比页至少能选中多个实验并触发对比状态变化

### 4) 去掉静默跳过式测试

以下模式不得继续作为“通过”逻辑存在：

- `if (await locator.isVisible(...).catch(() => false)) { ... }`

如果关键元素不存在，应：

- 直接失败
- 或在前置步骤中明确构造数据后再断言

不能“没看到元素就什么都不做然后测试通过”。

### 5) 如需测试数据准备，必须写清策略

由于 E2E 很可能依赖基础数据，你可以采用以下任一方式：

1. 在 `beforeAll` 里通过 API 准备测试数据
2. 使用固定的测试账号/种子数据
3. 在测试前调用已有脚本准备数据

要求：

- 报告里明确写出数据准备策略
- 不得隐含依赖“数据库里恰好已经有数据”
