## 一、任务背景

对 T53 汇报复核后，确认其方向部分正确，但“29 / 6 稳定基线”这一核心数字仍然失真：

1. T53 提供了两种视角：
   - 全局排除 `docs/tasks` 后得到 `0 / 0`
   - 在 `docs/tasks/M7` 下排除 `evidence` 与 `report.md` 后得到 `29 / 6`
2. 其中全局 `0 / 0` 视角可被独立复核。
3. 但按 T53 自己写出的第二条命令重新执行，当前真实结果并不是 `29 / 6`，而是更低的一组命中。
4. 进一步列出命中来源后可见，当前被该规则实际统计到的命中主要来自部分历史任务单文件，而数量并没有达到 T53 声称的 `29 / 6`。
5. 这说明 T53 当前失败的根因不是“跨轮次边界思路错误”，而是：
   - 仍然没有把“到底哪些文件被算入对象集合”以可复核方式固定下来；
   - 仍然依赖对正则规则效果的口头描述，而不是直接展示被统计文件集合与全量命中清单；
   - 因而再次把错误总数写进了汇报与证据。

本轮任务不再设计新的过滤哲学，不再扩写“业务视角 vs 治理视角”的解释框架，只处理一件事：

1. 用固定对象集和全量命中清单把 T53 的第二条统计口径彻底钉死，避免再次靠正则想象对象范围。

4. 让 T53 关于 `docs/tasks/M7` 视角的数字只依赖这套固定对象集证据，而不是口头解释正则效果。

## 四、详细交付要求

### 4.1 固定对象集必须显式落盘

必须至少提供以下其中一种证据：

1. 被统计文件路径完整清单；或
2. 被统计文件数量 + 每个文件的命中数明细；或
3. 一份可复跑的“先列文件、后检索”的两阶段证据，能让任何人确认哪些文件进入了统计集合。

要求：

1. 不允许只给正则规则，不给对象集结果。
2. 不允许继续仅凭“理论上 report/evidence 被排除了”来断言总数。
3. 必须让读者能看见哪些具体文件贡献了最终总数。

### 4.2 `docs/tasks/M7` 视角必须重新计数

对 T53 第二视角，也就是：

1. `docs/tasks/M7` 范围内
2. 排除 `evidence`
3. 排除 `report.md`

必须重新计算两组关键词的真实命中数，并与固定对象集证据一一对应。

### 4.3 全局 `0 / 0` 视角若保留，必须与第二视角严格分离

如果继续保留“全局业务视角 `0 / 0`”，必须明确写明：

1. 这是另一套统计范围；
2. 不能拿它为 `docs/tasks/M7` 视角的数字背书；
3. 两套视角的命令、范围、结论必须分栏呈现，禁止混写。

### 4.4 最终结论必须回答的问题

最终结论必须明确回答：

1. T53 的 `29 / 6` 为什么错。
2. 当前固定对象集实际包含哪些来源文件。
3. `docs/tasks/M7` 视角下两组关键词的真实总数是多少。
4. 全局 `0 / 0` 与 M7 历史任务单视角之间是什么关系。
