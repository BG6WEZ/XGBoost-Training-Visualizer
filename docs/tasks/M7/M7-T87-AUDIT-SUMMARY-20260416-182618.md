## 三、不通过项

### 1. `request_id` 未稳定注入到普通子 logger 的 JSON 日志中

`Task 4.4` 的关键要求是：

- 每条 API 请求日志包含 `request_id`

我实际在当前环境执行验证：

```bash
cd apps/api
& 'c:\Users\wangd\project\XGBoost Training Visualizer\.venv\Scripts\python.exe' -c "import os, io, logging; os.environ['LOG_FORMAT']='json'; from app.logging_config import setup_logging, get_logger, request_id_ctx; buf=io.StringIO(); setup_logging(); root=logging.getLogger(); root.handlers[0].stream=buf; token=request_id_ctx.set('req-123'); get_logger('demo.child').info('hello'); request_id_ctx.reset(token); print(buf.getvalue().strip())"
```

实际输出中的业务日志为：

```json
{"timestamp":"...","level":"INFO","message":"hello","module":"<string>","logger":"demo.child","function":"<module>","line":1}
```

其中**没有**：

- `request_id`

这说明当前把 `RequestIdFilter` 仅挂在 root logger 上，并不能稳定覆盖普通子 logger 发出的日志记录。

结论：

- “每个 API 请求日志包含 request_id”这一核心通过条件当前未真正达成

### 2. 报告样例与当前运行时事实不一致

`M7-T87-R` 报告中声称：

- JSON 日志样例包含 `request_id`
- API 请求日志通过 contextvars 自动注入 `request_id`

但当前实际运行时复测并未得到同样结果。

结论：

- 当前报告对 request_id 注入效果的描述超前于真实运行结果

### 3. 缺少“全量测试通过”的证据

`LAUNCH_DEVELOPMENT_PLAN.md` 中 `Task 4.4` 的通过条件明确要求：

- `全量测试通过`

但当前 `M7-T87-R` 报告中没有提供：

- 全量测试命令
- 全量测试输出

报告只提供了：

- JSON/text 输出样例

结论：
