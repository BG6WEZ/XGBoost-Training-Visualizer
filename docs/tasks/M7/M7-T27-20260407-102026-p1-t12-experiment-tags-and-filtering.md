## 一、任务背景

P1-S4 已完成结果分析和 Benchmark 指标统一输出，当前系统已经具备：

1. 实验创建、训练、结果查看、实验对比等基本链路；
2. 预测分析与 Benchmark 指标结构，可支持实验质量判断；
3. 但实验列表仍缺少更高层级的组织能力，用户无法通过标签和筛选快速定位实验。

P1-T12 的目标是补齐实验标签与筛选，使实验管理从“能查看”提升到“能组织、能检索、能快速定位”。

本轮只做标签与筛选闭环，不推进模型版本管理、回滚、认证或导出增强。

---

## 二、任务目标

1. 为实验增加标签能力，支持创建时或后续维护标签。
2. 实验列表支持按标签筛选。
3. 实验列表支持按日期范围和性能指标进行筛选或排序。
4. 前后端契约一致，标签字段命名统一、返回结构稳定。
5. 提供自动化测试和真实链路证据，证明组合筛选可用。

## 四、详细交付要求

### 4.1 实验标签能力

至少实现以下标签能力：

1. 每个实验可关联 0 到多个标签；
2. 标签采用明确的数据结构，建议为字符串列表；
3. 标签字段在创建实验、获取实验详情、获取实验列表时结构保持一致；
4. 标签值去重、去空格、去空字符串，避免脏数据扩散。

要求：

1. 后端返回结构中的标签字段命名统一；
2. 前端类型定义与后端一致；
3. 不允许前端自己临时拼接伪标签状态。

### 4.2 列表筛选能力

实验列表至少支持以下筛选维度：

1. 标签筛选；
2. 日期筛选（至少支持开始/结束时间范围）；
3. 性能指标筛选或排序（例如按 `val_rmse`、`r2`、`mae` 之一排序）。

要求：

1. 至少支持单条件筛选；
2. 至少支持 1 组组合条件筛选（例如标签 + 日期，或标签 + 性能）；
3. 筛选结果不能为空白崩溃，空结果需有明确提示。

### 4.3 API 契约与查询参数

后端应提供稳定的实验列表查询方式，可接受：

1. 扩展现有实验列表接口查询参数；
2. 或新增专用筛选接口，但必须说明与现有列表接口的边界。

若使用查询参数，建议至少覆盖：

- `tags`
- `date_from`
- `date_to`
- `sort_by`
- `sort_order`

要求：

1. 参数语义清晰；
2. 非法参数返回可读错误；
3. 文档和前端使用保持一致。

### 4.4 前端筛选交互

实验页至少补齐：

1. 标签展示；
2. 标签筛选入口；
3. 日期或排序筛选控件；
4. 清空筛选状态；
5. 空结果提示。

要求：

1. 不得破坏现有实验列表和训练入口；
2. 筛选状态变化后结果应真实更新；
3. 不得只做静态 UI，不接真实数据。

## 八、Copilot 审核重点

1. 是否只是加了前端标签展示，而后端列表查询并未真正支持筛选。
2. 标签字段是否在详情、列表、创建链路中结构一致。

3. 若日期和性能字段来源不稳定，排序结果可能与用户预期不一致，必须在汇报中说明口径。
