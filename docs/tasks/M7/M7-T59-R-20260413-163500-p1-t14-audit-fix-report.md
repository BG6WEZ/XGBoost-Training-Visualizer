## 一、修复的阻断项

### 阻断 1：数据库迁移未落地（已修复）

**问题：** `User` 模型新增 `must_change_password` 字段，但迁移链停在 `004_add_model_versions.sql`，无对应迁移文件。

**修复：** 新增 `apps/api/migrations/005_add_must_change_password.sql`
- 使用 `ALTER TABLE users ADD COLUMN IF NOT EXISTS` 确保幂等
- 默认值为 `FALSE`，不影响已有用户
- 对 admin 用户设置 `must_change_password = TRUE`

---

### 阻断 2：`/api/auth/me` 契约未同步（已修复）

**问题：** `GET /api/auth/me` 的 `UserResponse` 构造未传入 `must_change_password`，schema 默认值 `False` 会静默覆盖真实值。

**修复：** `apps/api/app/routers/auth.py` 中 `get_me()` 现在传入 `must_change_password=current_user.must_change_password or False`

---

### 阻断 3：前端首次改密最小闭环未实现（已修复）

**问题：** `LoginPage.tsx` 登录成功后直接 `navigate('/')`，不处理 `must_change_password`。

**修复：**
- `apps/web/src/lib/api.ts`：`UserResponse` 接口新增 `must_change_password?: boolean`
- `apps/web/src/contexts/AuthContext.tsx`：新增 `mustChangePassword` 状态，`login()` 返回 `must_change_password` 结果
- `apps/web/src/pages/LoginPage.tsx`：
  - 登录后若返回 `must_change_password=true`，显示强制改密表单
  - 改密表单阻止用户跳过，必须修改密码后才能进入首页
  - 改密成功后调用 `navigate('/')` 进入正常流程
