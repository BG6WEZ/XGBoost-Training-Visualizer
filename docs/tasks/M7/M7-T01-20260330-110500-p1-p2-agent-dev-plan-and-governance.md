## 一、任务目标

在 MVP（M0-M4）已通过验收的基础上，形成并执行 P1、P2 的完整研发计划。本任务定义：

1. P1、P2 的分阶段任务列表（可直接执行）。
2. 每个任务的交付物、验收标准、测试要求。
3. GitHub Copilot 与 Trae 的协作边界与工作机制。
4. 任务发布原则与汇报原则，确保全流程可追踪、可审计、可复核。

---

## 二、双AI协同原则（必须遵守）

### 原则 1：职责单向清晰，决策闭环可追溯

- GitHub Copilot 负责：任务发布、范围控制、质量门禁、证据审计、通过/驳回结论。
- Trae 负责：按任务实现代码、自测验证、提交证据、风险上报。
- 任何“通过”结论必须由 Copilot 基于证据给出，Trae 无权自判“最终通过”。

### 原则 2：证据优先于结论，事实优先于进度

- 没有命令输出证据的“完成”一律视为未完成。
- 未执行测试不得写“测试通过”，只能写“未验证”。
- 若进度与质量冲突，优先保证质量闭环，不得以赶进度跳过门禁。

---

## 三、任务发布原则（Copilot）

1. 每个子任务必须包含：任务编号、范围边界、修改文件范围、验收标准、测试命令、风险约束。
2. 任务粒度必须控制在 0.5-1.5 天可闭环，禁止一次发大包任务。
3. 每 1-2 个任务必须设置停点，等待 Trae 报告并审核后再下发下一批。
4. 禁止越里程碑开发：未通过当前任务不得提前进入后续任务。
5. 任务单必须明确“禁止修改项”（如映射文档、无关目录、无关依赖）。
6. 发布任务时必须声明：
   - 通过项
   - 未通过项
   - 风险项
   - 回滚点

---

## 四、任务汇报原则（Trae）

1. 汇报必须按模板提交，至少包含：
   - 已完成任务编号
   - 修改文件清单
   - 实际执行命令
   - 实际输出结果
   - 未验证项
   - 风险与限制
2. 测试结果必须区分：通过 / 跳过 / 未运行 / 失败。
3. 发现范围外问题可报告，但不得擅自修复范围外内容。
4. 每次汇报后必须进入“待审核”状态，不得连续自推进。
5. 若无法验证，必须写清阻塞原因、影响范围、替代验证方案。

## 五、P1 开发计划（建议 6 个 Sprint）

> 节奏建议：每个 Sprint 1 周；每周拆为 4-6 个任务；每任务独立验收。

### P1-S1：特征工程核心能力落地

#### P1-T01 时间/滞后/滚动特征后端能力
- 范围：API + Worker
- 交付物：
  - 特征工程配置 schema
  - worker 生成时间特征、lag、rolling 逻辑
  - 输出工件写入 workspace
- 验收标准：
  - 支持 hour/dayofweek/month/is_weekend
  - 支持 lag=[1,6,12,24] 与 rolling=[3,6,24]
  - 异常配置返回 422 + 错误原因
- 最小测试：
  - 单元测试：特征生成函数
  - 集成测试：API 发起任务 -> worker 完成 -> 工件落盘

#### P1-T02 预处理能力（缺失值/编码）
- 交付物：前向填充、均值填充、one-hot、label encoding
- 验收标准：
  - 不同策略在配置层可切换
  - 输出字段与 schema 同步
- 最小测试：
  - 参数化单测（多策略）
  - 合同测试（response 字段一致）

#### P1-T03 前端特征工程页面
- 交付物：/features/:datasetId 页面 + 预览
- 验收标准：
  - 可配置并提交特征任务
  - 展示生成特征数量与示例
- 最小测试：
  - 组件测试
  - E2E：从数据集页进入特征页并提交成功

### P1-S2：数据质量与多表融合

#### P1-T04 数据质量评分引擎
- 交付物：完整性/准确性/一致性/分布四维评分
- 验收标准：输出总分(0-100) + 问题列表 + 修复建议
- 最小测试：异常值、缺失值、重复值、时间连续性用例

#### P1-T05 质量报告页面
- 交付物：/data/:id/quality
- 验收标准：评分卡、问题清单、建议区块可视
- 最小测试：接口 mock 与真实接口联调各 1 轮

#### P1-T06 多表 Join（主表+天气/元数据）
- 交付物：Join 配置与执行能力
- 验收标准：Join 键可配置；Join 后行数变化可解释
- 最小测试：
  - 主表+天气表 join 成功
  - 键缺失时报错可读

### P1-S3：训练能力增强

#### P1-T07 Early Stopping + 参数模板
- 交付物：early_stopping_rounds；保守/平衡/激进模板
- 验收标准：
  - 提前停止可触发并记录 best_iteration
  - 模板切换后参数完整回填
- 最小测试：
  - 训练过程集成测试
  - 前端参数回填测试

#### P1-T08 参数校验与冲突提示
- 交付物：前后端双重校验
- 验收标准：非法参数阻断提交；提示具体字段
- 最小测试：负向测试 >= 8 条

#### P1-T09 并发训练与队列可视化
- 交付物：并发上限配置、排队状态展示
- 验收标准：至少 2 个任务并发 + 队列任务可观测
- 最小测试：并发场景 E2E + 队列长度一致性测试

### P1-S4：结果分析深化

#### P1-T10 预测 vs 实际 + 残差分析
- 交付物：散点图、残差直方图、Q-Q 图
- 验收标准：完成训练后页面可渲染全部图表
- 最小测试：
  - API 返回图表数据结构合同测试
  - 前端渲染快照测试

#### P1-T11 Benchmark 模式（ASHRAE/BDG2）
- 交付物：统一输出 RMSE/MAE/MAPE/R2
- 验收标准：不同数据集输出结构一致
- 最小测试：benchmark 集成测试 + 结果字段校验

### P1-S5：实验与模型管理

#### P1-T12 实验标签/筛选
- 交付物：标签管理、筛选器
- 验收标准：按标签/日期/性能过滤正确
- 最小测试：查询条件组合测试

#### P1-T13 模型版本管理
- 交付物：版本列表、版本比较、回滚
- 验收标准：训练完成自动建版本；可回滚并可追踪
- 最小测试：回滚链路测试 + 数据一致性测试

#### P1-T14 配置/报告导出
- 交付物：JSON/YAML 配置导出；PDF/HTML 训练报告
- 验收标准：导出文件可打开、字段完整
- 最小测试：导出文件结构校验

### P1-S6：认证与技术债清理

#### P1-T15 登录与用户管理（简化版）
- 交付物：登录、用户创建/禁用、密码重置
- 验收标准：管理员权限边界正确
- 最小测试：鉴权中间件测试 + 角色测试

#### P1-T16 技术债清理
- 交付物：修复现存失败测试、路径一致性、接口契约对齐
- 验收标准：主测试集全绿（允许已声明 skip）
- 最小测试：
  - pytest 全量
  - 前端 type-check
  - 关键 E2E 冒烟

## 七、统一测试要求（所有任务必须满足）

1. 单元测试：新增核心逻辑必须覆盖。

若命令无法执行，必须写明原因与替代验证。

### 7.3 通过判定

- 功能通过：满足任务验收标准。
- 质量通过：无新增 P0/P1 缺陷。
