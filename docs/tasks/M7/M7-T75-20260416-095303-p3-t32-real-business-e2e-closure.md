## 四、必须完成的最小工作

### 1) 去掉所有静默跳过式通过逻辑

以下模式本轮一律不允许继续存在于关键测试路径：

- `if (await xxx.isVisible(...).catch(() => false)) { ... }`
- `const ok = await locator.isVisible(...).catch(() => false); if (ok) { ... }`
- “找不到元素就直接截图并结束测试”

要求：

- 关键元素不存在时必须直接失败
- 或在前置步骤中先构造出必须存在的数据，再做强断言
- 不得把“没执行到关键动作”计为测试通过

### 2) 资产流程必须变成真实主路径

至少落成以下链路：

1. 登录进入系统
2. 实际上传一个 CSV 文件
3. 实际注册一个数据集
4. 注册成功后跳转到详情页
5. 详情页中对数据集名称、字段信息或关键元数据做真实断言

禁止：

- 只检查上传按钮是否可见
- 只调用文件输入但不验证结果
- 直接 `goto('/assets/1/quality')` 代替真实流程完成

### 3) 实验流程必须变成真实主路径

至少落成以下链路：

1. 基于真实存在的数据集打开创建实验页面
2. 实际填写必要字段
3. 实际提交创建
4. 在实验列表中验证新实验出现
5. 实际触发训练提交动作
6. 对提交成功状态、任务排队状态或页面状态变化做真实断言

禁止：

- 只验证“创建按钮可见”
- 只验证“表单里至少有一个输入框”
- 只验证“表格可能存在”

### 4) 结果流程必须绑定真实实验数据

至少落成以下链路：

1. 使用前面真实创建并提交过训练的实验
2. 打开结果页或详情页
3. 对指标、图表、状态字段、对比选择状态等做真实断言
4. 对实验对比页执行“选择多个实验并触发状态变化”的真实动作

禁止：

- 直接 `goto()` 固定 URL 后只断言 `body` 可见
- 找不到实验链接时无失败直接结束
- 只截图不验证数据

### 5) 测试数据准备策略必须显式、可重复

本轮必须明确采用以下一种策略，且写入报告：

1. 在 `beforeAll` / helper 中通过 API 准备测试数据
2. 使用固定种子数据，并验证其确实存在
3. 调用已有准备脚本生成测试所需数据

要求：

- 不得隐含依赖“数据库里刚好已有合适数据”
- 不得依赖人工临时改库后不在报告中说明
- 报告中必须写清每类流程使用了什么数据准备方式

### 6) 必须真实执行 Playwright，并附本轮原始输出

至少执行：

```bash
cd apps/web
npx playwright test
```

要求：

- 本轮报告中附完整原始输出
- 输出必须对应当前代码，不得继续引用旧轮次结果
- 若仍有失败，必须诚实记录失败项，不得写“已完成”
