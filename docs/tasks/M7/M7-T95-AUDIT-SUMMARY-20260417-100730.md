## 三、阻断项

### 1. `Task 5.2` 的硬门槛仍未满足

总计划 `Task 5.2` 的通过条件仍是：

- `所有端点满足 P95 目标`
- `无 5xx 错误`
- `输出性能基线报告`

本轮虽然满足了后两项，但第一项仍然没有满足。

### 2. 当前仅 1 / 5 端点达标

根据 `M7-T95-R`：

- `/health`：`2270ms` vs 目标 `< 50ms`，`FAIL`
- `/api/auth/login`：`4384ms` vs 目标 `< 500ms`，`FAIL`
- `/api/datasets/`：`717ms` vs 目标 `< 200ms`，`FAIL`
- `/api/experiments/`：`607ms` vs 目标 `< 200ms`，`FAIL`
- `/api/datasets/upload`：`1896ms` vs 目标 `< 3000ms`，`PASS`

也就是说，当前只是从 `0 / 5` 提升到 `1 / 5` 达标，距离计划门槛仍有明显差距。

### 3. “WSL2 是主因”目前仍是合理假设，不是已完成验证的结论

`M7-T95-R` 将主要根因归因为：

- Windows -> WSL2 -> Docker 的多层网络栈
- WSL2 的 CPU / I/O 调度开销

这个判断**有合理性**，但当前仓库证据仍只覆盖：

- Windows + Docker Desktop + WSL2 环境下的 benchmark

尚未提交：

- 原生 Linux 环境下的同口径复测
- 与 WSL2 的一一对照数据

因此目前最多只能说：

- **环境因素假设成立且值得继续验证**

不能据此直接把 `Task 5.2` 判为通过。

### 4. 报告中仍有一个目标值笔误，需要后续注意

`M7-T95-R` 的“未达标项”中写到：

- `/api/experiments/` P95 `< 500ms`

但总计划 `Task 5.2` 的目标是：

- `/api/experiments/` P95 `< 200ms`

本轮主结论不受该笔误影响，因为无论按 `200ms` 还是 `500ms`，当前都未通过；但后续收口时必须统一口径，避免再次出现报告与计划不一致。
